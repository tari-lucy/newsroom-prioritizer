# Newsroom Prioritizer

![CI](https://github.com/tari-lucy/newsroom-prioritizer/actions/workflows/ci.yml/badge.svg)

Сервис приоритизации инфоповодов для новостной редакции. Автоматически собирает новости
из открытых лент, отсекает чужой регион и дубли, оценивает вероятность «залетит»
(наберёт просмотры на сайте) и показывает выпускающему редактору ленту по убыванию
приоритета. Решает боль редакции: при ограниченном времени и штате помогает быстро отобрать
инфоповоды, которые дадут максимум охвата, и одной кнопкой подготовить рерайт под стиль издания.

Заказчик и первый пользователь — региональная телекомпания (Севастополь/Крым).

## Демо

1. Редактор открывает витрину — **ленту инфоповодов, отсортированную по вероятности «залетит»**.
2. Каждая карточка: класс приоритета (🔴 высокая / 🟡 средняя / ⚪ низкая), источник, гео-теги.
3. Кнопкой **«🔄 Запустить сбор»** можно собрать свежие инфоповоды прямо сейчас (иначе сбор идёт по расписанию).
4. На карточке — **👍/👎** (копится как обучающий сигнал) и **«✍️ Рерайт»** (готовит уникальный текст под стиль редакции).
5. Раздел **«Источники»** — добавить/выключить ленту через интерфейс.

## Архитектура

```
             nginx ─► api (FastAPI) ── скоринг in-process (модель) — быстро
                          │
 scheduler ─► коннекторы (RSS→соцсети…) ─► гео-фильтр ─► полный текст ─► дедуп ─► PostgreSQL
                          │
     webview (Streamlit) ◄┘  лента, 👍/👎, рерайт, управление источниками
                          │
        рерайт ─► RabbitMQ ─► worker (LLM) — медленно, вне HTTP
```

| Сервис | Роль |
|---|---|
| `api` (FastAPI) | REST API: приём, лента, скоринг, постановка рерайта |
| `web-proxy` (nginx) | реверс-прокси, единая точка входа |
| `webview` (Streamlit) | витрина редактора |
| `scheduler` | периодический автосбор инфоповодов |
| `worker` | генерация рерайта из очереди |
| `trainer` | периодическое переобучение приоритизатора |
| `rabbitmq` | очередь задач (медленный LLM вне HTTP-цикла) |
| `database` (PostgreSQL) | инфоповоды, рерайты, обратная связь |

**Пайплайн:** `сбор → гео-фильтр → полный текст → дедуп → скоринг → лента → рерайт`. Источники
абстрагированы как **коннекторы** (`Connector.fetch()`): подключены RSS и паблики ВКонтакте
(VK API), новые типы (Telegram и т.п.) добавляются классом-коннектором без изменения остального
пайплайна. Регион вынесен в `config/region.yml`
(не хардкод). RSS отдаёт только анонс, поэтому перед дедупом и скорингом сервис дотягивает
полный текст статьи по ссылке (`trafilatura` + `cloudscraper`) — так пайплайн согласован с тем,
что модель обучена на полных текстах.

## Стек

FastAPI, Streamlit, PostgreSQL (SQLModel), RabbitMQ (pika), nginx, Docker Compose,
scikit-learn (скоринг и дедуп), OpenAI-совместимый API (рерайт через vsellm).

## Быстрый старт

```bash
cp .env.example .env        # значений по умолчанию достаточно для запуска
docker compose up --build
```

Больше ничего не нужно — сервис поднимется и **сразу работает**. Приоритизатор при этом считает
на прозрачной эвристике-заглушке; чтобы он работал на **обученной модели**, выполните
необязательный шаг [«Обученная модель»](#обученная-модель-необязательно) перед запуском.

После старта:
- Витрина редактора: http://localhost:8501
- REST API (Swagger): http://localhost/docs
- Healthcheck: http://localhost/health
- Панель RabbitMQ: http://localhost:15672

Вход в витрину (демо-редактор создаётся автоматически): **логин `editor`, пароль `editor123`**.
Доступ к API защищён JWT — токен выдаёт `POST /auth/login`.

## Обученная модель (необязательно)

Модель приоритизатора версионируется через DVC и лежит в публичном хранилище — её можно
подтянуть, даже если DVC у вас не установлен и вы с ним не работали. Без этого шага сервис
тоже полностью работает, просто приоритизатор считает на эвристике-заглушке вместо модели.

**1. Установите DVC** любым удобным способом:

```bash
sudo snap install dvc --classic                 # Ubuntu/Debian — проще всего
```
или без sudo, в отдельное окружение:
```bash
python3 -m venv .dvcenv && .dvcenv/bin/pip install "dvc[http]"
```
> Обычный `pip install "dvc[http]"` на свежих Debian/Ubuntu может выдать ошибку
> `externally-managed-environment` — тогда используйте snap или venv выше.

**2. Подтяните модель** (в папке репозитория, до `docker compose up`):

```bash
dvc pull -r public                 # если ставили через venv: .dvcenv/bin/dvc pull -r public
```

Появится файл `config/virality_model.joblib` — сервис подхватит его автоматически.

## Переменные окружения

Все настройки — в `.env` (шаблон с комментариями: `.env.example`). Ключевые:

| Переменная | Назначение |
|---|---|
| `REGION`, `REGION_CONFIG` | активный регион и файл гео-терминов |
| `DEDUP_THRESHOLD` | порог косинуса для дублей |
| `INGEST_INTERVAL_MINUTES` | период автосбора |
| `MODEL_PATH` | путь к обученной модели приоритизатора |
| `LLM_BASE_URL`, `LLM_API_KEY`, `LLM_MODEL` | подключение LLM рерайта (vsellm). Без ключа — заглушка |
| `TEXTRU_API_KEY` | проверка уникальности рерайта (Text.ru). Без ключа — пропускается |
| `METRIKA_TOKEN`, `METRIKA_COUNTER` | источник меток (просмотры) для переобучения. Без токена петля простаивает |
| `VK_TOKEN` | сервисный токен VK API для сбора из пабликов ВКонтакте. Без токена vk-источники пропускаются |

## Приоритизатор и метрики

Задача — ранжирование инфоповодов по вероятности попасть в топ по просмотрам (3 класса
приоритета: низкая / средняя / высокая). Целевая переменная — относительная (топ-20% внутри
месяца публикации), что снимает дрейф трафика.

Валидация — скользящее окно по времени (out-of-time). Главная метрика — **PR-AUC** (классы
несбалансированы ~1:4, ROC-AUC при дисбалансе завышает); операционная — **Precision@top20%**
(доля хитов среди отобранных), ординальная — **QWK**.

| Модель | PR-AUC | Precision@top20% |
|---|---|---|
| случайная (база) | 0.203 | ×1.0 |
| эвристика по ключевым словам | 0.332 | ×1.53 |
| **TF-IDF + LogReg (итоговая)** | **0.446** | **×2.04** |
| BERT (news_tg_rubert) | 0.43–0.44 | ×1.9–2.05 |

Три семейства моделей (TF-IDF, эмбеддинги, BERT) сходятся у ~0.44 → потолок в данных, а не в
модели. Итоговая — TF-IDF+LogReg: выигрывает главную метрику и в разы дешевле (работает на CPU
в контуре редакции). Отбор по модели поднимает долю хитов в **≈2 раза** относительно базовой.

Модель подключается в одной точке — `backend/pipeline/scoring.py` (`Scorer`), поэтому её замена
не затрагивает API и витрину. Если файла модели нет, `Scorer` работает на прозрачной
эвристике-заглушке, и сервис остаётся полностью рабочим.

### Переобучение (continuous training)

Сервис `trainer` (`backend/trainer/`) периодически переобучает приоритизатор:

```
дозревшие метки (просмотры из Метрики) → обучение кандидата
   → out-of-time гейт (не хуже текущей на holdout) → промоут артефакта в MODEL_PATH
```

Метки берутся из Яндекс.Метрики (просмотры по URL) → относительный топ внутри месяца публикации,
как в исходном моделировании. Ключевой нюанс — **задержка метки**: «залетело ли» известно только
через ~месяц (просмотры накапливаются), поэтому учимся не на свежих текстах, а на дозревших
(`LABEL_MATURATION_DAYS`).
Гейт защищает от деградации на шумном месяце — новая модель заменяет текущую, только если не
хуже по PR-AUC. Ценность петли — адаптация к дрейфу (новая лексика/повестка), а не рост точности.
Обновлённую модель `Scorer` подхватывает при перезапуске.

## Рерайт

Медленная генерация вынесена в воркер (`backend/worker/`) через очередь. Модель подключается
через vsellm (OpenAI-совместимый API) и выбирается переменной `LLM_MODEL` без правки кода;
без ключа работает заглушка. Промпт задаёт стиль редакции и запрет на выдумывание фактов.
Уникальность готового текста проверяется через Text.ru (контроль качества рерайта); процент
дописывается к результату отдельно, не задерживая показ текста редактору.


## Версионирование модели (DVC)

Артефакты модели версионируются через DVC: в git — только указатели `*.dvc`, сами файлы
хранятся в DVC-remote. Текущая модель — `config/virality_model.joblib`. Каждая новая версия
(в т.ч. из петли переобучения) добавляется `dvc add` — история версий модели отделена от кода.

Настроены два remote к одному хранилищу на сервере:

- `storage` (SSH) — для заливки артефактов владельцем: `dvc push`;
- `public` (HTTP, только чтение) — чтобы модель мог подтянуть любой: `dvc pull -r public`
  (как установить DVC — см. раздел [«Обученная модель»](#обученная-модель-необязательно)).

Если модель не подтянута, приоритизатор работает на прозрачной эвристике-заглушке.

## Тесты

```bash
pip install -r backend/requirements-dev.txt
cd backend && pytest
```

Тесты идут на изолированной sqlite-БД, без внешних зависимостей. CI (GitHub Actions) на каждый
push прогоняет линтер, тесты и сборку docker-образов.
